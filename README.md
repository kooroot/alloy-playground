# alloy-playground

End-to-end playground for the Rust [`alloy`](https://alloy.rs/) crate, wrapped
in a Bun + Vite + TanStack web UI. Built to verify every dApp primitive —
wallet connect, balance/network read, ETH transfer (EIP-1559), ERC-20
deploy/transfer/balanceOf, live Transfer-event subscription, and
server-side key generation — through a single cohesive stack.

> **Hybrid architecture.** The backend is **pure alloy**. The frontend uses
> **viem strictly as a browser EIP-1193 wallet adapter** — every chain read
> and broadcast still funnels through the Rust service. No wagmi, no
> ethers.js, no `createPublicClient`. One source of truth per concern.

## At a glance

| What | Where | Notes |
|---|---|---|
| Read account balance + nonce + chain tip | `GET /api/account/{addr}`, `GET /api/network` | page: `/` |
| Switch Anvil ↔ Sepolia at runtime | `POST /api/network` | no backend restart |
| Send ETH (server-signed) | `POST /api/tx/eth/send-local` | `.env PRIVATE_KEY` path |
| Send ETH (MetaMask) | `POST /api/tx/eth/build` + `eth_sendTransaction` | browser signs |
| Send raw tx | `POST /api/tx/eth/send` | viem `signTransaction` path |
| Deploy DemoToken (ERC-20) | `POST /api/erc20/deploy` | page: `/erc20` |
| ERC-20 transfer (both modes) | `POST /api/erc20/transfer/{send-local,build}` | — |
| `balanceOf` | `GET /api/erc20/balance/{holder}` | — |
| Live Transfer events | `ws://.../ws/erc20/transfers` | page: `/events` |
| Mint a random private key | `POST /api/wallet/new` | alloy `PrivateKeySigner::random()` |
| Mint a 12-word BIP-39 mnemonic + derive key | `POST /api/wallet/new-mnemonic` | alloy `MnemonicBuilder::<English>` |
| Restore key from mnemonic (BIP-44) | `POST /api/wallet/from-mnemonic` | index-aware |
| Hot-swap the live server signer | `POST /api/wallet/use` | page: `/wallet` |

## Architecture

```
┌──────────────────────────┐   REST + WS    ┌──────────────────────────┐
│  web/ (Vite + React)     │ ─────────────▶ │  backend/ (axum + alloy) │
│  TanStack Router + Query │   via /api     │  PrivateKeySigner        │
│  zustand, viem (EIP-1193)│   via /ws      │  http + ws providers     │
└──────────────────────────┘                └──────────┬───────────────┘
                                                       │ JSON-RPC / WS
                                                       ▼
                                              ┌──────────────────┐
                                              │ anvil or Sepolia │
                                              └──────────────────┘
```

- Backend exposes an OpenAPI spec (`GET /api/openapi.json`, generated by
  `utoipa`). `bun run codegen` turns it into `web/src/types/api.d.ts` so the
  TypeScript fetch client is always in lockstep with the Rust handlers.
- Vite dev server proxies `/api` and `/ws` to `127.0.0.1:8080`, which keeps
  the frontend origin-less (no CORS) and lets Vite's port fallback work
  transparently (3000 → 3001 → …).

## Phases

Each phase is self-contained and leaves a working artifact. You can stop at
any phase and the earlier ones stay usable.

### Phase 1 — Backend skeleton + read-only queries

- **Files.** `backend/src/main.rs`, `state.rs`, `routes/{health,network,account}.rs`
- **Endpoints.** `GET /api/health`, `GET /api/network`, `GET /api/account/{addr}`
- **What it proves.** axum boots, `AppState` wraps a `DynProvider` behind a
  tokio `RwLock` (so Phase 2 can hot-swap the network), `alloy` can read
  `chain_id`, latest block, balance, and nonce.
- **Exit criterion.** `curl localhost:8080/api/account/0xf39F…` returns the
  anvil balance.

### Phase 2 — Web scaffold + codegen + toggles

- **Files.** `web/src/main.tsx`, `routeTree.ts`, `routes/__root.tsx`,
  `routes/index.tsx`, `lib/store.ts`, `lib/api.ts`, `scripts/codegen.ts`
- **UI.** Landing page `/` with address/balance/chainId/latest-block tiles +
  a `NetworkWalletToggles` header (zustand-backed, persisted to
  `localStorage`). Network toggle POSTs `/api/network` and invalidates
  dependent queries.
- **Typed API client.** `openapi-fetch` over the generated `paths` type;
  `unwrap(await api.GET(...))` is the idiom used everywhere.
- **Exit criterion.** Page shows address + balance + chainId + latest
  block; toggles survive reload.

### Phase 3 — ETH transfer, both signing modes

- **Files.** `backend/src/routes/tx.rs`, `web/src/routes/transfer.tsx`,
  `web/src/lib/metamask.ts`
- **Endpoints.**
  - `POST /api/tx/eth/send-local` — server signs with `PrivateKeySigner`
    and broadcasts.
  - `POST /api/tx/eth/build` — returns a fully-populated EIP-1559 tx
    (pending-nonce, `estimate_eip1559_fees` with a 1-gwei priority floor,
    `estimate_gas`) as hex strings the browser hands straight to
    `window.ethereum.request({ method: "eth_sendTransaction" })`.
  - `POST /api/tx/eth/send` — broadcast a raw signed tx (viem
    `signTransaction` path, kept for comparison).
- **UI.** `/transfer` flips between the two mutations on the wallet
  toggle. MetaMask path: Connect button → `eth_requestAccounts` →
  viem `requestAddresses()`.
- **Exit criterion.** Local-signer path verified end-to-end on anvil.
  MetaMask path ships and renders a Connect button, but the popup itself
  requires a real MetaMask extension in a real browser.

### Phase 4 — ERC-20 deploy + transfer + balanceOf

- **Files.** `backend/src/routes/erc20.rs`, `contracts.rs`,
  `contracts/src/DemoToken.sol`, `web/src/routes/erc20.tsx`
- **Solidity → Rust bindings.** `alloy::sol! { #[sol(rpc)] DemoToken,
  "contracts/out/DemoToken.sol/DemoToken.json" }` — Foundry emits the
  artifact, `sol!` generates typed `deploy_builder`, `transferCall`,
  `balanceOf`, and the `Transfer` event type.
- **Endpoints.** `deploy`, `transfer/send-local`, `transfer/build`,
  `GET balance/{holder}?token=0x…`. Each falls back to the last-deployed
  address when `token` is omitted, so you don't have to round-trip it
  through the UI.
- **UI.** `/erc20` has three cards: Deploy, Transfer (branches on wallet
  mode exactly like `/transfer`), balanceOf query.
- **Exit criterion.** Full loop (deploy → transfer → balance goes up on
  recipient, down on sender) passes on anvil. The math is exact to the
  wei, verified across 10+ transfers.

### Phase 5 — WebSocket Transfer event stream

- **Files.** `backend/src/routes/events_ws.rs`, `web/src/routes/events.tsx`,
  `state.rs` (adds `ws_provider: Option<DynProvider>`)
- **Backend.** Upgrades `/ws/erc20/transfers` to a WebSocket, subscribes
  via `ws_provider.subscribe_logs(&Filter::new().address(token)
  .event_signature(DemoToken::Transfer::SIGNATURE_HASH)
  .from_block(Latest))`, and forwards each decoded event as a JSON frame
  tagged `hello | transfer | warn | error`.
- **Frontend.** Native `WebSocket` + reconnect (exp backoff, 1s → 10s
  cap), 50-event ring buffer (reducer), fatal frames skip the retry loop.
  Vite `/ws` proxy forwards to the Rust service so the browser URL is
  always same-origin.
- **Exit criterion.** Transfer hits the UI within one anvil block
  (~2 s). Verified across bursts of 6+ concurrent transfers.

### Phase 6 — alloy-native wallet demo

- **Files.** `backend/src/routes/wallet.rs`, `state.rs` (adds
  `swap_signer()`), `web/src/routes/wallet.tsx`,
  `web/src/components/WalletBadge.tsx`
- **What it disproves.** "alloy can't do wallets because the popup
  doesn't appear." alloy is a Rust crate, not a browser app — it mints
  keys server-side and has no GUI. This page surfaces that in the UI:
  - `PrivateKeySigner::random()` → random 32-byte key
  - `Mnemonic::<English>::new_with_count(&mut OsRng, 12)` +
    `MnemonicBuilder::<English>::default().phrase(p).build()` →
    BIP-39 12-word mnemonic + derived key at `m/44'/60'/0'/0/0`
  - `swap_signer(PrivateKeySigner)` rebuilds the HTTP signing provider
    against the same network URL without a process restart.
- **Smoking gun.** Feeding `"test test test ... test junk"` (the
  Foundry/Hardhat default test phrase) to `/api/wallet/from-mnemonic`
  reproduces the exact anvil #0 address and private key — proving the
  BIP-44 derivation is correct, not just random bytes.
- **Safety.** The endpoint echoes raw private keys and mnemonics back to
  the browser in plaintext. That is a **deliberate learning trade-off**.
  Do not copy this surface into anything that touches real funds.

### Phase 7 — UX polish (not a planned phase; added mid-session)

- Disconnect button + `wallet_revokePermissions` (EIP-2255) →
  `WalletBadge.tsx`
- Overview page gets a "Connect MetaMask" CTA when wallet mode is
  `metamask` and no address is cached (previously the Overview page
  rendered no connect trigger at all — a UX dead-end)
- Vite binds to `127.0.0.1` (not `localhost`) so the IPv4 strictPort
  fallback actually triggers on macOS

## Quickstart

### Prereqs

```bash
curl -fsSL https://bun.sh/install | bash
curl -L https://foundry.paradigm.xyz | bash && foundryup
# Rust: https://rustup.rs
# Optional: brew install just mprocs  (Justfile path)
```

### First-time setup

```bash
bun run install:all                       # root + web deps
cp backend/env.example backend/.env       # edit for Sepolia if needed
(cd backend/contracts && forge build)     # emits contracts/out/DemoToken.sol/DemoToken.json
                                          # which alloy::sol! reads at cargo build
```

### Run everything

```bash
bun run dev
```

Spawns `anvil + backend + web` under `concurrently` with labeled streams
(~3 s cold start):

- <http://localhost:3000> — web UI
- <http://localhost:8080/api/openapi.json> — OpenAPI spec (try `/swagger` in
  a REST client)
- <http://127.0.0.1:8545> — anvil JSON-RPC

One at a time: `bun run dev:anvil`, `bun run dev:backend`, `bun run dev:web`.

### Alternative: `just` + `mprocs` (3 split panes)

```bash
brew install just mprocs
just dev
```

Both paths run the same three commands.

### Regenerating typed API client

After any change to backend routes/schemas (backend must be running):

```bash
bun run codegen
```

### Verify

```bash
bun run check   # cargo check + tsc --noEmit
bun run test    # cargo test — spawns anvil, drives the full ERC-20 flow over HTTP
bun run build   # cargo build + vite build
```

`bun run test` runs `backend/tests/erc20_flow.rs`, which spawns a fresh
anvil via `alloy::node_bindings::Anvil`, builds the real `AppState` +
`Router` against it, and walks `/api/health` → `/api/network` →
`/api/account` → `/api/erc20/deploy` → `/api/erc20/transfer/send-local`
→ `/api/erc20/balance` over reqwest. Requires `foundryup` (for `anvil`).

## Manual E2E smoke

Order matters — Events tab must be open **before** you fire a transfer, or
you'll miss the subscription window:

1. **Deploy.** Visit `/erc20` → click **Deploy DemoToken**. The backend
   remembers the address.
2. **Subscribe.** Open `/events` in a new tab. Status pill flips to
   `LIVE · 0x… · chain 31337` within ~1 s.
3. **Fire.** Back on `/erc20`, send a transfer (any amount, wallet = Local).
   The event lands on `/events` within a single block (~2 s on anvil).
4. **MetaMask path.** Flip wallet toggle to `MetaMask`, click
   **Connect wallet** on `/transfer` or `/erc20`, approve, send. MetaMask
   pops up, approves, broadcasts. Event stream picks it up like any other
   Transfer.
5. **Wallet mint.** `/wallet` → **Generate 12-word mnemonic** → **Activate
   on server**. Now `/transfer` (Local mode) signs as a brand-new address
   with zero balance — trying to send ETH returns `Insufficient funds`,
   which is the strongest proof the hot-swap is real. Restore via the
   junk mnemonic (`test test ... junk`) to get anvil #0 back.

## Sharp edges

- **Sepolia WS.** Phase 5 needs a WebSocket RPC. Most free Sepolia
  endpoints are HTTP-only; grab a WS-capable one from Alchemy / Infura /
  QuickNode free tier and put it in `SEPOLIA_WS_URL`.
- **MetaMask `maxPriorityFeePerGas` on Sepolia.** Occasionally comes back
  absurdly low and stalls the tx. Backend floors it at 1 gwei in
  `/api/tx/eth/build` (applied to `/api/erc20/transfer/build` too).
- **Overview has no Connect button by design.** Read-only page. The CTA
  lives on `/`, `/transfer`, and `/erc20` once the wallet mode toggle is
  set to MetaMask. Phase 7 added the `/` CTA so this isn't a dead-end.
- **OpenAPI codegen drift.** After any backend schema change, the frontend
  types don't auto-update. `bun run codegen` (backend running) re-fetches.
- **alloy pinned to 2.0 loosely.** `Cargo.toml` uses `alloy = "2.0"`. If
  behavior drifts on CI, pin exact (`=2.0.1`) — minor releases of alloy
  have been known to rename items.
- **Never push a real `.env`.** `.gitignore` blocks `backend/.env*`. The
  checked-in `backend/env.example` uses anvil's well-known default
  account #0 key — safe for local testing, **never** for real networks.

## Not shipped

Honestly tracked so you know what you're NOT getting:

- **Phase 0 — Codex second-opinion review.** The plan called for handing
  the design to `codex:rescue` for an independent architecture pass
  before code. The codex runtime crashed the session twice; abandoned.
- **Integration test.** The plan mentioned a ~80-LOC `cargo test` using
  `alloy::node_bindings::Anvil::new().spawn()` to cover deploy + transfer
  + balance in-process. Not written. `bun run check` (cargo check + tsc)
  is the only automated guard today.
- **Sepolia manual verification.** All 6 phases were runtime-verified on
  anvil. Sepolia path is wired but not exercised end-to-end with a real
  MetaMask extension + faucet ETH — that requires user action.
- **`accountsChanged` sync.** If you revoke the site permission from the
  MetaMask extension UI directly (not our Disconnect button), our
  `connectedAddress` store stays stale until the next manual Connect.
  A ~10-line `useEffect` listening to `eth.on("accountsChanged", …)`
  would fix it.
